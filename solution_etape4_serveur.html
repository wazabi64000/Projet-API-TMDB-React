1️ Objectif

Lancer le serveur Express avec sécurisation et middleware :

  • Helmet pour sécuriser les headers
  • CORS pour gérer les requêtes cross-origin
  • express-rate-limit pour limiter les requêtes
  • dotenv pour la configuration

Action : Créez le fichier server.js ou app.js

Métaphore : Helmet = casque de sécurité, Rate limit = vigile, CORS = panneau d’autorisation.

2️ Explications détaillées

Chaque middleware protège ou configure le serveur :

  • Helmet : ajoute des headers sécurisés
  • CORS : définit qui peut accéder à l’API
  • express-rate-limit : limite le nombre de requêtes pour éviter les abus
  • dotenv : charge les variables d’environnement (port, clé API)

Action : Installer et configurer chaque middleware avant de démarrer le serveur.

3️ Exemple concret


import express from 'express';
import helmet from 'helmet';
import cors from 'cors';
import rateLimit from 'express-rate-limit';
import dotenv from 'dotenv';

dotenv.config();
const app = express();

// Middleware
app.use(helmet());
app.use(cors());
app.use(express.json());
app.use(rateLimit({ windowMs: 15*60*1000, max: 100 }));

// Route test
app.get('/', (req, res) => res.send('Serveur OK'));

// Lancer serveur
const PORT = process.env.PORT || 3000;
app.listen(PORT, () => console.log(`Serveur démarré sur ${PORT}`));

Action : Démarrez le serveur et testez http://localhost:3000

4️ Cas concrets / erreurs fréquentes


// Port déjà utilisé
Error: listen EADDRINUSE: address already in use

Solution : changer le port dans .env ou arrêter le processus existant.


// Middleware mal importé
import helmet from 'helmet';
TypeError: helmet is not a function

Solution : vérifier que helmet est installé et correctement importé.

5️ Bonnes pratiques

  • Middleware global en haut avant les routes
  • Limiter le nombre de requêtes pour chaque route critique
  • Ne jamais exposer directement vos clés dans le code

Métaphore : Chaque middleware est comme un agent de sécurité : placer au bon endroit est crucial.
